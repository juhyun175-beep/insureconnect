### 발견사항(구조 개선 권고 — 이번 PR 미수정)

1. **입금 게이트 없음**: `ad_orders.status`가 `paid`로 전이되는 코드가 없다(‘결제완료’ 라벨은 도달 불가). 승인만으로 자동이행되므로, 입금 전 오승인 시 **카톡 전회원 알림은 되돌릴 수 없다**. 이번 PR에서 승인 확인창에 입금대기 경고를 추가했지만, 근본적으로는 관리자 "입금확인" 액션(→ status='paid')을 만들고 paid 이전 승인 시 카톡 발송만 보류하는 상태전이를 권장.
2. **카톡 일괄발송 스케일 한계**: 현재 수신동의 283명. 발송이 승인 요청 안에서 동기 실행되며(5 동시), 회원당 fetch 1~2회 → Cloudflare 무료 플랜은 요청당 서브리퀘스트 50개 제한으로 **약 25~50명 이후 전원 실패**, 유료(1000개)도 약 500명 규모에서 한계. `functions/api/admin/kakao-broadcast.js`는 순차 발송이라 283명 기준 응답까지 40~80초. 권장: 발송 큐 테이블 + cron(외부 트리거) 배치(예: 회당 80명) 처리로 전환. 성장 전 필수 과제.
3. **유료 추천공고(18,000원)와 무료 첫승인 맛보기가 동일한 3일**: 최초 승인 시 누구나 featured 3일 무료가 적용되므로, 첫 공고에서 추천공고 옵션을 산 구매자는 **추가 이득이 0**이다(포인트 상단노출은 50P에 7일이라 역전됨). 유료 옵션을 +7일(또는 무료 3일에 가산)로 차별화 권장.
4. **환불 시 이행 롤백 없음**: 환불 승인해도 dm_enabled/홈배너 캠페인/featured가 남는다. 수기 환불 정책이라 치명적이진 않지만, 환불 시 체크리스트(배너 캠페인 비활성 등)라도 관리자 화면에 안내 권장.

## 3. 보안 — 수정 6건 / 권고 4건

### 이번 PR에서 수정 (상세 CHANGELOG)
- **[높음] 공개 API 개인정보 노출**: 채용/강의 목록 GET이 등록자 이름·연락처를 비인증 반환, 단건 GET은 상태 무관 `SELECT *`(연락처·쿠폰 필드 포함) 공개 → 관리자 전용으로 게이트. 회귀 테스트 추가.
- **[중간] 업로드 남용**: user-upload 서버측 레이트리밋 없음 → IP당 20건/시간.
- **[중간] 오승인 비가역 발송**: 승인 확인창 입금대기 경고.
- **[낮음] 포인트 이중지불 race**, **[낮음] cron 시크릿 타이밍 비교**, **[정보] 죽은 Supabase anon key 하드코딩 3개 파일** 제거.

### 유지해도 되는 항목(권고)
- **관리자 인증 구조**: 정적 `x-admin-secret` + constant-time 비교 + sessionStorage 보관. 1인 운영엔 합리적. 단 verify 외 관리자 엔드포인트에는 실패 지연이 없으므로 **시크릿은 충분히 길게(24자+) 유지**하고 유출 의심 시 즉시 교체(`wrangler pages secret put ADMIN_SECRET`).
- **CORS `Access-Control-Allow-Origin: *`** 전 응답 공통: 헤더 기반 인증이라 실익 악용은 어려우나, 관리자 응답은 자체 도메인으로 좁히면 더 안전.
- **/api/track/*** 무인증 쓰기: 봇 필터 + 120일 자동정리 있음. D1 쓰기 남용 시 Turnstile 고려.
- **/api/files GET**: R2 버킷 전체가 공개 서빙 전제 — 버킷에 비공개 파일을 절대 넣지 말 것.
- 카카오 OAuth(state CSRF), 세션(해시 저장·httpOnly·Lax), DM 접근통제(서버 owner 도출), 관리자 회원 API(토큰 미노출·LIKE 이스케이프), SQL 파라미터 바인딩 전반: **양호**.

## 4. 성능(버벅임) 분석 — 구조 요인이 커서 로드맵 제안

측정(라이브): index.html **682KB**(미니파이, brotli 전송 152KB), admin.html **399KB**. index 인라인 JS ~392KB·CSS ~318KB, 정적 DOM ~2,700 노드.

원인 정리:
1. **단일 파일 아키텍처**: JS/CSS가 전부 HTML 인라인 → 브라우저가 **매 방문(캐시 60초)마다 682KB 재다운로드+재파싱**. 콘텐츠 1글자 변경에도 전체 무효화. 중급 모바일에서 첫 페인트 지연·입력 버벅임의 주원인.
2. **admin 10초 폴링**: 대시보드가 보이는 동안 10초마다 `/api/stats` + 재렌더(전체 innerHTML 갱신). 30~60초로 완화 권장.
3. 홈 60초×2 인터벌, 채팅 폴링(2.5s, 열려있을 때) 등은 탭 비활성 시 정지 처리가 이미 잘 되어 있음.

권장 로드맵(효과 순):
1. 인라인 JS/CSS를 `app.[hash].js`/`app.[hash].css`로 외부화 + 1년 immutable 캐시(_headers에 이미 규칙 존재) → 재방문 전송량 ~150KB→수 KB.
2. 등록 모달·뷰어·카드뉴스 등 거대 기능 블록 lazy 로드(현재도 32개 script 블록으로 나뉘어 있어 분리 용이).
3. admin 폴링 주기 완화 + 목록 diff 렌더.
4. (이번 PR로) 고아 이미지 3.4MB 제거 — 배포 아티팩트/저장소 경량화.
